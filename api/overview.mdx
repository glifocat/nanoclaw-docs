NanoClaw is a single Node.js host process that orchestrates per-session agent containers. Channel adapters land platform messages, the router resolves them through an entity model (users → messaging groups → agent groups → sessions), and the agent-runner inside each container processes them by reading and writing two SQLite databases.

## Architecture

### Core components

- **Orchestrator** (`src/index.ts`) — initializes the central DB, runs migrations, starts channel adapters, delivery polls, and the host sweep
- **Inbound router** (`src/router.ts`) — thread policy, messaging group lookup, sender resolution, fan-out, and engage evaluation
- **Session manager** (`src/session-manager.ts`) — resolves sessions, opens `inbound.db` and `outbound.db`, manages the heartbeat path
- **Container runner** (`src/container-runner.ts`) — spawns per-agent-group containers with session and outbox mounts; calls `onecli.ensureAgent`
- **Container runtime** (`src/container-runtime.ts`) — Docker vs Apple Container selection and orphan cleanup
- **Delivery** (`src/delivery.ts`) — polls `outbound.db` and dispatches to channel adapters and registered system-action handlers
- **Host sweep** (`src/host-sweep.ts`) — 60s loop for `processing_ack` sync, stale detection, due-message wake, and recurrence advancement
- **Modules** (`src/modules/`) — pluggable hooks: permissions, scheduling, agent-to-agent, approvals, self-mod, typing
- **Channels** (`src/channels/`) — adapter registry and Chat SDK bridge; specific adapters install from the `channels` branch

### Key concepts

#### Entity model

```
users (id "<channel>:<handle>", kind, display_name)
user_roles (user_id, role, agent_group_id)         — owner | admin (global or scoped)
agent_group_members (user_id, agent_group_id)

agent_groups (workspace, container config, provider)
    ↕ many-to-many via messaging_group_agents (engage_mode, sender_scope, session_mode, ...)
messaging_groups (one chat/channel on one platform; unknown_sender_policy)

sessions (agent_group_id + messaging_group_id + thread_id → per-session container)
```

Privilege is user-level (owner/admin), not channel-level. Wirings carry the engage policy, sender scope, and session mode for each (messaging group, agent group) pair.

#### Message flow

1. The channel adapter receives a message and calls `onInbound` on the router
2. Thread policy collapses `threadId` for non-threaded platforms
3. The router resolves the messaging group (auto-creating on mention/DM), then fans out to wired agents
4. Each engaging agent gets a write to its session's `inbound.db` and a container wake
5. The agent-runner polls `inbound.db`, calls the configured provider, and writes responses to `outbound.db`
6. The host's delivery poll reads `outbound.db` and dispatches via the adapter's `deliver()` callback

#### Sessions and the two-DB split

Each session has two SQLite files under `data/v2-sessions/<agent_group_id>/<session_id>/`:

- `inbound.db` — host writes, container reads (`messages_in`, routing, destinations, pending_questions)
- `outbound.db` — container writes, host reads (`messages_out`, `processing_ack`, `session_state`)

Exactly one writer per file — no cross-mount lock contention. The heartbeat is a file touch at `/workspace/.heartbeat`, not a DB update.

## Database

The central database (`data/v2.db`) holds everything that isn't per-session:

- `users`, `user_roles`, `agent_group_members` — entity model
- `agent_groups`, `messaging_groups`, `messaging_group_agents` — workspaces and wirings
- `sessions` — per-session status and container state
- `agent_destinations`, `pending_approvals`, `pending_sender_approvals`, `pending_channel_approvals` — module state
- `chat_sdk_*` — Chat SDK bridge state
- `schema_version` — managed by `src/db/migrations/`
