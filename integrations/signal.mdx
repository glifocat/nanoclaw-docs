Add Signal as a messaging channel for NanoClaw. The adapter speaks JSON-RPC to a [signal-cli](https://github.com/AsamK/signal-cli) TCP daemon — no Chat SDK bridge, only Node.js builtins.

Signal has no bot API. NanoClaw registers either as a full Signal account on a dedicated phone number (recommended) or as a secondary device linked to your existing number.

## Overview

The Signal channel is a native adapter installed by the `/add-signal` skill. It's copied from the `channels` branch into `src/channels/signal.ts` and self-registers via `src/channels/index.ts`. The host manages the signal-cli daemon lifecycle by default.

### Features

- **DMs and groups** — 1:1 chats and Signal GroupV2 conversations
- **Markdown formatting** — `**bold**`, `*italic*`, `` `code` ``, code fences, `~~strike~~`, `||spoiler||` translated to Signal's offset-based text styles
- **Quoted replies** — `replyTo*` fields populated from Signal quotes
- **Typing indicators** — DMs only (Signal doesn't support group typing)
- **Echo suppression** — outbound messages matched on `(platformId, text)` within a 10s TTL to avoid syncMessage loops
- **Note to Self** — messages you send to your own account from another device route inbound with `isFromMe: true`
- **Voice attachments** — detected but not transcribed by default; the agent receives a `[Voice Message]` placeholder. Run `/add-voice-transcription` for local transcription via parakeet-mlx

Not yet supported: outbound file attachments, edit/delete messages, reactions.

## Prerequisites

### Java 17+

signal-cli requires Java 17 or newer:

```bash
java -version
```

If missing:

- **macOS:** `brew install --cask temurin@17`
- **Debian/Ubuntu:** `sudo apt-get install -y default-jre`
- **RHEL/Fedora:** `sudo dnf install -y java-17-openjdk`

Java 17 through 25 all work.

### signal-cli

- **macOS:** `brew install signal-cli`
- **Linux:** download the native binary from [GitHub releases](https://github.com/AsamK/signal-cli/releases):

```bash
SIGNAL_CLI_VERSION=$(curl -fsSL https://api.github.com/repos/AsamK/signal-cli/releases/latest | python3 -c "import sys,json; print(json.load(sys.stdin)['tag_name'][1:])")
curl -fsSL "https://github.com/AsamK/signal-cli/releases/download/v${SIGNAL_CLI_VERSION}/signal-cli-${SIGNAL_CLI_VERSION}-Linux-native.tar.gz" \
  | tar -xz -C ~/.local
ln -sf ~/.local/signal-cli ~/.local/bin/signal-cli
signal-cli --version
```

<Note>
The Linux native tarball extracts a single binary directly to `~/.local/signal-cli` (not into a subdirectory). The symlink puts it on PATH.
</Note>

## Adding Signal

Use the `/add-signal` skill in Claude Code:

```bash
claude
/add-signal
```

The skill is idempotent — every step is safe to re-run. It fetches the `channels` branch, copies `src/channels/signal.ts` and its tests into your fork, appends the self-registration import, and runs the build. No npm packages to install — the adapter uses only Node.js builtins.

## Registration

You have two paths. The new-number path is recommended and battle-tested.

### Path A: Register a new number (recommended)

Use a dedicated SIM or VoIP number. NanoClaw owns it entirely.

<Warning>
Signal requires SMS verification before voice. Some VoIP providers are blocked even for voice calls. If registration fails with an auth error, try a different provider or a physical SIM.
</Warning>

<Steps>
  <Step title="Solve the CAPTCHA">
    Signal requires a CAPTCHA on first registration:

    1. Open `https://signalcaptchas.org/registration/generate.html` in a browser
    2. Solve the captcha
    3. Right-click the **Open Signal** button → **Copy Link**
    4. The link starts with `signalcaptcha://` — the token is everything after that prefix
  </Step>

  <Step title="Request SMS verification">
    ```bash
    signal-cli -a +1YOURNUMBER register --captcha "PASTE_TOKEN_HERE"
    ```
  </Step>

  <Step title="Voice fallback (if SMS won't deliver)">
    Wait ~60 seconds after the SMS request, then:

    ```bash
    signal-cli -a +1YOURNUMBER register --voice --captcha "SAME_TOKEN"
    ```

    Signal calls and reads a 6-digit code. The same captcha token is reusable — no need to solve a new one.

    <Warning>
    You must request SMS first. Requesting voice immediately fails with `Invalid verification method: Before requesting voice verification…`.
    </Warning>
  </Step>

  <Step title="Verify the code">
    ```bash
    signal-cli -a +1YOURNUMBER verify CODE
    ```

    No output means success.
  </Step>

  <Step title="Set a profile name (optional)">
    <Warning>
    Stop NanoClaw before running signal-cli commands directly — the daemon holds an exclusive lock on its data directory while running.
    </Warning>

    ```bash
    # macOS
    launchctl unload ~/Library/LaunchAgents/com.nanoclaw.plist
    signal-cli -a +1YOURNUMBER updateProfile --name "YourBotName"
    # optionally: --avatar /path/to/avatar.jpg
    launchctl load ~/Library/LaunchAgents/com.nanoclaw.plist

    # Linux
    systemctl --user stop nanoclaw
    signal-cli -a +1YOURNUMBER updateProfile --name "YourBotName"
    systemctl --user start nanoclaw
    ```
  </Step>
</Steps>

### Path B: Link as a secondary device

Joins an existing Signal account as a secondary device. Simpler, but NanoClaw shares your personal number.

```bash
signal-cli -a +1YOURNUMBER link --name "NanoClaw"
```

This prints a `tsdevice:` URI. Render it as a QR code on your phone via **Settings → Linked Devices → Link New Device**. QR codes expire in ~30 seconds — re-run if it expires.

## Configuration

Add to `.env`:

```bash
SIGNAL_ACCOUNT=+1YOURNUMBER
```

### Optional settings

```bash
# TCP daemon host and port (default: 127.0.0.1:7583)
SIGNAL_TCP_HOST=127.0.0.1
SIGNAL_TCP_PORT=7583

# Path to the signal-cli binary (default: resolved on PATH)
SIGNAL_CLI_PATH=/usr/local/bin/signal-cli

# Whether NanoClaw manages the daemon lifecycle (default: true).
# Set to false if you run signal-cli daemon externally.
SIGNAL_MANAGE_DAEMON=true

# signal-cli data directory (default: ~/.local/share/signal-cli)
SIGNAL_DATA_DIR=~/.local/share/signal-cli
```

<Warning>
Keep `SIGNAL_TCP_HOST` on `127.0.0.1`. The signal-cli daemon has no auth — binding it to a public interface would expose your full Signal account to the network.
</Warning>

### Sync to container

```bash
mkdir -p data/env && cp .env data/env/env
```

## Wiring

### DMs

After the service starts, send any message to the Signal number from your personal Signal app. The router auto-creates a `messaging_groups` row. Then look it up:

```bash
sqlite3 data/v2.db \
  "SELECT id, platform_id FROM messaging_groups WHERE channel_type='signal' ORDER BY created_at DESC LIMIT 5"
```

Pass the `id` to `/init-first-agent` or `/manage-channels` to wire it to an agent group.

### Groups

Add the Signal number to a group from your phone, send any message, then wire the resulting row the same way. For isolated per-group sessions:

```bash
NOW=$(date -u +"%Y-%m-%dT%H:%M:%S.000Z")
sqlite3 data/v2.db "
INSERT OR IGNORE INTO messaging_group_agents
  (id, messaging_group_id, agent_group_id, session_mode, priority, created_at)
VALUES
  ('mga-'||hex(randomblob(8)), 'mg-GROUPID', 'ag-AGENTID', 'isolated', 0, '$NOW');
"
```

### Grant user access

New Signal users (including the owner's Signal identity) are silently dropped with `not_member` until granted access. After the user's first message appears in `messaging_groups`:

```bash
NOW=$(date -u +"%Y-%m-%dT%H:%M:%S.000Z")
sqlite3 data/v2.db "
INSERT OR REPLACE INTO user_roles (user_id, role, agent_group_id, granted_by, granted_at)
  VALUES ('signal:UUID', 'owner', NULL, 'system', '$NOW');
INSERT OR IGNORE INTO agent_group_members (user_id, agent_group_id, added_by, added_at)
  VALUES ('signal:UUID', 'ag-AGENTID', 'system', '$NOW');
"
```

Find the UUID from `messaging_groups.platform_id` or the `users` table.

## Channel info

| Field | Value |
|---|---|
| Channel type | `signal` |
| Terminology | Signal has "chats" (1:1 DMs) and "groups" |
| Threads | Not supported |
| Platform-id (DM) | `signal:{UUID}` — sender's Signal UUID (ACI), **not** their phone number |
| Platform-id (group) | `signal:{base64GroupId}` — base64-encoded GroupV2 ID |
| Default isolation | One agent per Signal account; mixed-operator groups should use `isolated` session mode |

## Troubleshooting

<AccordionGroup>
  <Accordion title="Daemon not reachable">
    ```bash
    grep "Signal" logs/nanoclaw.log | tail
    ```

    If you see `Signal daemon failed to start. Is signal-cli installed and your account linked?`:
    - Confirm `signal-cli` is on PATH (or set `SIGNAL_CLI_PATH`)
    - Confirm the account is linked: `signal-cli -a +1YOURNUMBER listIdentities` should succeed without prompting

    If you see `Signal daemon not reachable at 127.0.0.1:7583` and `SIGNAL_MANAGE_DAEMON=false`, start the daemon yourself:

    ```bash
    signal-cli -a +1YOURNUMBER daemon --tcp 127.0.0.1:7583
    ```
  </Accordion>

  <Accordion title="Bot not responding">
    1. Channel initialized: `grep "Signal channel connected" logs/nanoclaw.log | tail -1`
    2. Channel wired: `sqlite3 data/v2.db "SELECT mg.platform_id, mg.name FROM messaging_groups mg JOIN messaging_group_agents mga ON mg.id = mga.messaging_group_id WHERE mg.channel_type='signal'"`
    3. Service running:
       - macOS: `launchctl print gui/$(id -u)/com.nanoclaw`
       - Linux: `systemctl --user status nanoclaw`
  </Accordion>

  <Accordion title="Lost connection mid-session">
    `Signal channel lost TCP connection to signal-cli daemon` means the daemon dropped the connection. Restart the service to re-establish.
  </Accordion>

  <Accordion title="Messages dropped with not_member">
    The Signal user hasn't been granted membership. See [Grant user access](#grant-user-access) above. This affects every new Signal user, including the owner's Signal identity — that identity is a separate user record from their identity on other channels even if it's the same person.
  </Accordion>

  <Accordion title="Captcha required">
    Signal requires a captcha for new registrations. Open `https://signalcaptchas.org/registration/generate.html`, solve it, right-click **Open Signal**, copy the link, extract the token after `signalcaptcha://`.
  </Accordion>

  <Accordion title="Invalid verification method (voice)">
    `Invalid verification method: Before requesting voice verification…` means you skipped SMS. Request SMS first, wait ~60 seconds, then request voice. The same captcha token works for both steps.
  </Accordion>

  <Accordion title="Config file in use / daemon lock">
    signal-cli holds an exclusive lock on its data directory while the daemon is running. Stop NanoClaw before running any `signal-cli` commands directly, then restart afterward.
  </Accordion>

  <Accordion title="Group replies routing as DMs">
    Modern Signal groups use GroupV2. The adapter must extract the group ID from `envelope?.dataMessage?.groupV2?.id` — not `groupInfo?.groupId`, which is GroupV1/legacy. If group messages route as DMs, check `src/channels/signal.ts` and confirm the groupId extraction falls through to `groupV2.id`.
  </Accordion>

  <Accordion title="Java not found">
    Install Java 17 or newer — see [Prerequisites](#prerequisites).
  </Accordion>

  <Accordion title="QR code expired (Path B)">
    QR codes expire in ~30 seconds. Re-run the link command to generate a new one.
  </Accordion>
</AccordionGroup>
