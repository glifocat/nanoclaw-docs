## Overview

`ncl` is the admin CLI for NanoClaw's central database. Use it to inspect and modify agent groups, messaging groups, wirings, users, roles, members, destinations, sessions, approvals, dropped messages, and user DMs.

Two transports, one interface:

- **Host** — `ncl` connects via a Unix socket served by the host process (`src/cli/socket-server.ts`).
- **Container** — Agents call `ncl` from inside their session container; commands are framed through the session DB transport (`container/agent-runner/src/cli/ncl.ts`).

Read commands run inline. Writes from inside a container go through the [approval flow](#approval-flow) before executing.

## Usage

```
ncl <resource> <verb> [<id>] [--flags]
ncl <resource> help
ncl help
```

Flags use `--hyphen-case` (e.g. `--agent-group-id`) and are mapped to `underscore_case` DB columns automatically.

`list` supports filtering by any non-auto column and pagination via `--limit` (default `200`):

```bash
ncl wirings list --messaging-group-id mg_xyz
ncl sessions list --limit 50
```

## Resources

| Resource | Verbs | Description |
|----------|-------|-------------|
| `groups` | list, get, create, update, delete | Agent groups (workspace, personality, container config) |
| `messaging-groups` | list, get, create, update, delete | A single chat or channel on one platform |
| `wirings` | list, get, create, update, delete | Links a messaging group to an agent group (session mode, triggers, priority) |
| `users` | list, get, create, update | Platform identities (`<channel>:<handle>`) |
| `roles` | list, grant, revoke | Owner / admin privileges (global or scoped to an agent group) |
| `members` | list, add, remove | Unprivileged access gate for an agent group |
| `destinations` | list, add, remove | Where an agent group is allowed to send messages |
| `sessions` | list, get | Active sessions (read-only) |
| `user-dms` | list | Cold-DM cache (read-only) |
| `dropped-messages` | list | Messages from unregistered senders (read-only) |
| `approvals` | list, get | Pending approval requests (read-only) |

For composite-key resources (`roles`, `members`, `destinations`), use the custom verbs (`grant`/`revoke`, `add`/`remove`) instead of `create`/`delete`.

Run `ncl <resource> help` to see all available fields, types, enums, and which fields are required or updatable.

## Access rules

| Verbs | Access |
|-------|--------|
| `list`, `get` | Open — runs inline, no approval needed |
| `create`, `update`, `delete`, `grant`, `revoke`, `add`, `remove` | Approval-gated when called from inside a container |

Host callers run all commands inline. Container callers must go through admin approval for writes.

## Approval flow

When a container agent runs a write command, the dispatcher does not execute it directly. Instead:

<Steps>
  <Step title="Agent invokes a write">
    The container runs `ncl groups create --name "Research" --folder research`.
  </Step>
  <Step title="Dispatcher returns approval-pending">
    The command returns immediately with an `approval-pending` response. The action has **not** been executed.
  </Step>
  <Step title="Admin receives a card">
    `pickApprover` selects an admin or owner (preferring scoped admins for the agent group, then global admins, then owners) and `pickApprovalDelivery` DMs them an approval card showing the exact command and arguments.
  </Step>
  <Step title="Admin approves or rejects">
    On approve, the registered `cli_command` approval handler re-dispatches the original frame as a host caller and notifies the agent with the result. On reject, the agent receives a system message saying the request was rejected.
  </Step>
</Steps>

The agent does not poll or retry — the result arrives as a system message in the same conversation.

<Note>
A write returning `approval-pending` is not an error. The agent should wait for the follow-up system message containing the executed result.
</Note>

## Examples

### Read commands (no approval)

```bash
# List all agent groups
ncl groups list

# Get details for a specific group
ncl groups get abc123

# See field definitions for a resource
ncl wirings help

# List all wirings for a specific messaging group
ncl wirings list --messaging-group-id mg_xyz

# Show who is an owner or admin
ncl roles list
```

### Write commands (approval required from container)

```bash
# Create a new agent group
ncl groups create --name "Research" --folder research

# Update a group
ncl groups update abc123 --name "Research v2"

# Grant a global admin role
ncl roles grant --user telegram:jane --role admin

# Grant a scoped admin role for a specific agent group
ncl roles grant --user discord:bob --role admin --group abc123

# Add a member to an agent group
ncl members add --user-id telegram:jane --agent-group-id abc123

# Allow an agent group to send messages to a messaging group
ncl destinations add --agent-group-id abc123 --messaging-group-id mg_xyz
```

## When agents should use ncl

Inside a container, prefer `ncl` over guessing about NanoClaw configuration:

- **Look up your own config** — `ncl groups get <your-group-id>` to inspect agent group settings.
- **Find who you're wired to** — `ncl wirings list` to see which messaging groups route to which agent groups.
- **Check user roles** — `ncl roles list` to see who is an owner or admin.
- **Answer system questions** — when a user asks about groups, channels, users, or configuration, query `ncl` rather than guessing.

## How it works

| Component | Purpose |
|-----------|---------|
| `src/cli/socket-server.ts` | Unix socket server on the host; handles `ncl` invocations from outside containers |
| `src/cli/socket-client.ts` | Client used by the host `bin/ncl` wrapper |
| `container/agent-runner/src/cli/ncl.ts` | Container-side wrapper; frames requests through the session DB transport |
| `src/cli/dispatch.ts` | Transport-agnostic dispatcher; gates approval-required commands and registers the `cli_command` approval handler |
| `src/cli/crud.ts` | Generic CRUD helper for registering resource verbs |
| `src/cli/resources/` | Per-resource definitions (groups, wirings, roles, etc.) |
| `src/cli/registry.ts` | Command registry — every resource verb is looked up here |

Setup installs `ncl` to `/usr/local/bin/ncl` on the host and adds a `~/.local/bin/ncl` symlink. The container image places it on the agent's `PATH` automatically.
