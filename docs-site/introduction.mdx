NanoClaw is a personal AI assistant you can talk to from the apps you already use. It runs Claude agents in isolated Linux containers, keeps each agent's workspace explicit, and ships as a codebase small enough to inspect and fork.

Use this portal by role: start with the fastest path if you just want an assistant, read the configuration pages if you are wiring channels and agents, and use the deployment pages when you need to operate NanoClaw for a team.

## What you get

- **Agents in containers.** Each running conversation is isolated from your host and from other sessions unless you choose to share state.
- **Messaging apps as front doors.** WhatsApp, Telegram, Discord, Slack, Teams, iMessage, Matrix, Google Chat, Webex, Linear, GitHub, WeChat, and Resend email adapters install on demand with `/add-<channel>` skills.
- **Agent groups as workspaces.** An agent group owns instructions, memory, skills, mounts, provider settings, and the files under `groups/<folder>/`.
- **Explicit wirings.** A messaging group is a chat or channel. A wiring connects that messaging group to one or more agent groups and controls who can trigger them.
- **Secure credentials.** Agents do not hold raw API keys. Outbound requests route through OneCLI's Agent Vault, which injects credentials at request time and enforces per-agent policy.

## How the isolation model works

```text
messaging apps → host process (router) → inbound.db → container (Bun, Claude Agent SDK) → outbound.db → host process (delivery) → messaging apps
```

A single Node host orchestrates per-session agent containers. A message arrives, the host routes it through an entity model (user → messaging group → wiring → agent group → session), writes it to the session's `inbound.db`, and wakes the container. The agent-runner inside the container polls `inbound.db`, runs Claude, and writes responses to `outbound.db`. The host polls `outbound.db` and delivers back through the channel adapter.

Two SQLite files per session, each with exactly one writer — no cross-mount lock contention, no IPC, no stdin piping. See [Architecture overview](/architecture) and [Database and sessions](/database-and-sessions) for the full picture.

<Tip>
If you remember one distinction, make it this: **messaging groups are chats; agent groups are workspaces/identities; wirings connect them.** Sharing a messaging app does not automatically share an agent's memory or files.
</Tip>

## Design principles

- **Small enough to understand.** One process, a few source files, no microservices. Ask Claude Code to walk you through the codebase.
- **Secure by isolation.** Agents run in Linux containers and see only what's mounted.
- **Built for the individual.** NanoClaw is meant to be forked. You make your own copy and have Claude Code shape it to your needs, instead of configuring a generic system.
- **Customization = code changes.** No configuration sprawl. Want different behavior? Change the code — the codebase is small enough that this is safe.
- **Skills over features.** Trunk ships the registry and infrastructure. Channels live on a long-lived `channels` branch; alternative agent providers (OpenCode, Codex, Ollama) live on `providers`. You run `/add-telegram`, `/add-opencode`, and so on, and the skill copies exactly the module you need into your fork.
- **Best harness, best model.** NanoClaw uses Claude Code via Anthropic's Claude Agent SDK by default; other providers are drop-in options, configurable per agent group.
